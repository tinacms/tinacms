# Next.js Kitchen-Sink

Next.js 15 App Router + React 18 site with TinaCMS visual editing, block-based pages, rich-text content, and multi-collection schemas. Uses Tailwind CSS 4 and Biome.

## Architecture

- **App Router** — pages live under `app/`. Each route has a server component (`page.tsx`) that runs at build time via `generateStaticParams`, plus a client component (`client-page.tsx`) that calls `useTina()` for live editing. Server queries the generated Tina client; the client component re-uses the returned `data`/`query`/`variables` as props.
- **Static generation with ISR** — every list/detail page sets `export const revalidate = 300` (5-minute ISR). `generateStaticParams` enumerates content via `client.queries.*Connection()`.
- **Block-based pages** — follows the shared block dispatcher pattern (see [examples/AGENTS.md](../../AGENTS.md)); implementation-specifics below.
- **Admin UI** — served by a Next.js rewrite: `/admin` → `/admin/index.html`, generated by `tinacms build` into `public/admin/`.

## Coding Standards

Repo-wide Biome + strict TypeScript apply (see root [AGENTS.md](../../../AGENTS.md)). Shared TinaCMS patterns (block dispatcher, `useTina()`/`tinaField()`, Tailwind theme, image sanitisation, admin UI selectors, slugify, GraphQL quirks) live in [examples/AGENTS.md](../../AGENTS.md). App-specific deltas:

- `lint`/`format` use Biome.
- Path alias `@/*` maps to the project root (not `./src`). Import like `@/components/blocks`, `@/tina/__generated__/client`.
- `next.config.js` enables `optimizePackageImports` for `react-icons` and `tinacms`.
- Image `remotePatterns` allow localhost only — when introducing new image hosts, update the list.

Collection schemas live in [tina/collections/](tina/collections/).

## Key Patterns

### App Router server/client split

Server component (`app/*/page.tsx`) runs the Tina query and passes `data`/`query`/`variables` through to a client component (`app/*/client-page.tsx`) that calls `useTina()`:

```tsx
// client-page.tsx
const { data } = useTina({
  query: props.query,
  variables: props.variables,
  data: props.data,
});
return (
  <>
    <h1 data-tina-field={tinaField(data.blog, 'title')}>{data.blog.title}</h1>
    <div data-tina-field={tinaField(data.blog, '_body')}>
      <TinaMarkdown components={customComponents} content={data.blog._body} />
    </div>
  </>
);
```

See [app/blog/[filename]/client-page.tsx](app/blog/[filename]/client-page.tsx) and [app/[...urlSegments]/client-page.tsx](app/[...urlSegments]/client-page.tsx) for full examples.

### Block dispatcher specifics

[components/blocks/index.tsx](components/blocks/index.tsx) keeps a `typeNameMap` of GraphQL `__typename` → short template name (`PageBlocksHero` → `hero`, etc.), then `switch`es on the template to `dynamic()`-import the block component. **Legacy `PageBlockPageBlocks*` typenames are mapped for backwards compatibility — keep those entries when editing.**

Block schemas live in `tina/schemas/blocks.ts`; shared field schemas (dates, actions, colours, `makeSlugify`) in `tina/schemas/shared-fields.ts`.

### Static params

```tsx
export async function generateStaticParams() {
  const allPages = await client.queries.pageConnection({ first: 1000 });
  return (allPages.data?.pageConnection.edges ?? [])
    .map((edge) => ({ urlSegments: edge?.node?._sys.breadcrumbs || [] }))
    .filter((x) => x.urlSegments.length >= 1)
    .filter((x) => !x.urlSegments.every((s: string) => s === 'home'));
}
```

From [app/[...urlSegments]/page.tsx](app/[...urlSegments]/page.tsx). The `home` filter prevents `home.md` from double-rendering — it's served from `app/page.tsx` directly.

Config lives at [tina/config.tsx](tina/config.tsx); sanitisation helpers at [lib/utils.ts](lib/utils.ts).

## Routes

| Route | Source | Description |
|-------|--------|-------------|
| `/` | `app/page.tsx` | Home page — queries `home.md` |
| `/[...urlSegments]` | `app/[...urlSegments]/page.tsx` | Block-based pages (`about`, `services`, etc.) |
| `/blog` | `app/blog/page.tsx` | Blog listing |
| `/blog/[filename]` | `app/blog/[filename]/page.tsx` | Blog detail — rich-text body, hero, author, dates |
| `/authors` | `app/authors/page.tsx` | Author listing |
| `/authors/[filename]` | `app/authors/[filename]/page.tsx` | Author detail |
| `/posts` | `app/posts/page.tsx` | Post listing |
| `/posts/[...urlSegments]` | `app/posts/[...urlSegments]/page.tsx` | Post detail (supports nested paths) |
| `/admin` | rewrite → `public/admin/index.html` | TinaCMS admin panel |

## E2E Tests

Playwright suite in [e2e/](e2e/). App-specific:

- `e2e/*.spec.ts` — frontend: `home`, `blog`, `posts`, `authors`, `navigation`, `edge-cases`. Fully parallel.
- `e2e/admin/*.spec.ts` — admin-panel CRUD: `author`, `blog`, `post`, `page`. Serial within each collection.
- [playwright.config.ts](playwright.config.ts): `timeout: 60s`, `expect.timeout: 30s`, `baseURL: http://localhost:3000`, Chromium-only, CI retries 2 / local 0, webServer auto-starts `pnpm dev`.

## Gotchas

- **TODO** in [components/markdown-components.tsx](components/markdown-components.tsx): `NewsletterSignup` is a stub — no backend wired up.
- **Legacy typenames** — the `typeNameMap` in [components/blocks/index.tsx](components/blocks/index.tsx) has duplicate entries for `PageBlockPageBlocks*` to preserve legacy inline-edit metadata. Don't remove them when refactoring.
- **`_template` on blocks** is runtime metadata used by TinaCMS inline-edit mode; it's not in the generated schema types, so access via the dispatcher's runtime switch rather than typed field access.
